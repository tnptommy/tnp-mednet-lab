# Change Management

## Purpose

This document defines the change management process for TNP MedNet.

## Change Types

| Type | Description | Example |
|---|---|---|
| Standard Change | Pre-approved low-risk change | Documentation update |
| Normal Change | Requires review and approval | BGP neighbour change |
| Emergency Change | Urgent change to restore service | Firewall fix during outage |

## Change Approval Matrix

| Risk Level | Approval Required |
|---|---|
| Low | IT Operations Manager |
| Medium | CTO and IT Operations Manager |
| High | CTO, IT Operations Manager, and Risk/Compliance Director |
| Emergency | IT Operations Manager approval, retrospective review required |

## Change Request Template

| Field | Description |
|---|---|
| Change title | Short description |
| Business reason | Why the change is needed |
| Systems affected | Devices, VLANs, sites, users |
| Risk level | Low, medium, high, emergency |
| Implementation plan | Step-by-step change process |
| Rollback plan | Steps to reverse the change |
| Test plan | Validation after implementation |
| Approvers | Required sign-off |
| Change window | Planned date and time |

## CAB Process

1. Change request is submitted.
2. Technical impact is reviewed.
3. Business impact is reviewed.
4. Risk and compliance impact is reviewed.
5. Approval is granted or rejected.
6. Change is implemented.
7. Testing is completed.
8. Documentation is updated.
9. Change record is closed.

## Emergency Change Process

Emergency changes may be approved quickly if clinical access or security is at risk. A retrospective review must be completed after the incident.
